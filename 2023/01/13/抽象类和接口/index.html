普通类（class）：只有具体实现；

抽象类（abstract）：具体实现和规范（抽象方法）都有；

接口（interface）：只有规范，没有具体实现。

1. 抽象类（abstract class）

1.1. 抽象类的定义

抽象类就是类的抽象，将类进行模板化。通过abstract定义规范和约束，然后要求子类来定义具体实现。通过抽象类，我们就可以做到严格限制子类的设计，使子类之间更加通用。

抽象类也是类，内部可以包含普通方法和属性，甚至构造方法。

  • 抽象方法：只有方法的定义，没有方法体的实现，这种方法就是抽象方法。抽象方法是没有方法体的，也就是没有大括号{}。
  • 抽象类：只要类被abstract修饰了，就是抽象类；一个类中如果有抽象方法，那么这个类就必须是抽象类，必须加上abstract关键字。

1.2. 抽象类的使用

1.3. 抽象类的特点

  • 抽象类不能直接实例化对象，只是一个约束，只能靠子类去实现它。
  • 抽象类里面可以既有普通方法，也有抽象方法；也可以只有普通方法，或只有抽象方法。
  • 抽象方法一定只存在于抽象类中，因为有抽象方法的类一定是抽象类。
  • 抽象方法不能是private的。
  • 抽象方法不能被final和static修饰，因为抽象方法要被子类重写。
  • 抽象类必须被继承，并且继承后子类要重写父类的抽象方法，否则子类也是抽象类，必须要使用abstract修饰。

2. 接口（interface）

2.1. 接口的定义

接口就是比“抽象类”还“抽象”的“抽象类”，可以更加规范的对子类进行约束。彻底实现了：规范和具体实现的分离。（规范做接口，就是架构师来做；具体实现是比较低端的工作，即所谓“面向接口编程”）

  • 接口就是规范，定义的是一组规则，体现了现实世界中“如果你是…则必须能…”的思想；
  • 接口的本质是约束和规范，就像契约与法律，制定之后就必须遵守。
  • Object Orientation的精髓就是，如何对对象进行抽象。讨论设计模式，都只针对应用那些具备了抽象能力的语言（c++，java， c#等）。

2.2. 接口的使用

接口定义关键词：interface；

接口实现类关键词：implements。

2.2.1. 定义接口1

接口Animals，接口中所有定义都是抽象的public abstract，这两个修饰词可省略。

2.2.2. 定义接口2

接口Animals，接口中所有定义都是抽象的public abstract，这两个修饰词可省略。

2.2.3. 定义实现类

实现类CreatureImpl，接口都需要有实现类，命名以Impl结尾，这个是implements的缩写，一般写业务的具体实现方式。

  • IDEA的快捷键ctrl + i。

  • 实现接口的类必须重写接口的所有方法。

  • java是单继承，但是可通过接口实现多继承。

2.3. 接口的特点

  • 定义一个约束或者规范；
  • 一个约束用于多个实现类（具体实现方式有差别，即所谓重写接口方法）；
  • 接口的方法都是：public abstract；
  • 接口的属性都是：public static final，即所谓常量；
  • 接口不能直接被实例化，必须用其实现类来实例化；因为接口中没有构造方法。
  • implements实现多个接口，即”伪多继承“。

3. 继承和接口

继承：”is a“，由下往上继承。

接口：”has a“，由上往下规范。

4. 抽象类和接口

接口的设计目的，是对类的行为进行约束（更准确的说是一种“有”约束，因为接口不能规定类不可以有什么行为，“如果你是…则必须能…”，当然你还可以会其他的），也就是提供一种机制，可以强制要求不同的类具有相同的行为。它只约束了行为的有无，但不对如何实现行为进行限制。结合泛型，理解“接口为何是约束”。

抽象类的设计目的，是代码复用。当不同的类具有某些相同的行为（记为行为集合A），且其中一部分行为的实现方式一致时（A的非真子集，记为B），可以让这些类都派生于一个抽象类。在这个抽象类中实现了B，避免让所有的子类来实现B，这就达到了代码复用的目的。而A减B的部分，留给各个子类自己实现。正是因为A-B在这里没有实现，所以抽象类不允许实例化出来（否则当调用到A-B时，无法执行）。

类1有行为A，B，C；类2有行为A, B, D, E；且类1和类2的A, B实现方式完全一致，可以将类1和类2派生于一个抽象类。在这个抽象类中实现A，B，避免在类1和类2中重复实现A，B，以达到代码复用。剩余的C，以及D和E由各子类独立实现。
